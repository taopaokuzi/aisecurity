## 2. 产品背景

企业员工申请系统权限时，通常需要经历填表、领导审批、IT 开通、测试验证等多个环节，平均耗时 3 到 5 天。流程中的常见问题包括：

- 员工不会准确描述需要什么权限，申请内容模糊。
- 审批人只理解业务需求，不理解具体系统权限项。
- IT 需要把业务语言翻译成系统 scope、角色或资源权限，沟通成本高。
- 权限开通后经常出现“开多了、开少了、开错了”的情况，导致反复确认。
- 权限到期后缺乏有效治理，容易出现长期残留权限。

随着 Agent 在企业业务中的使用增加，传统“人申请、人审批、人配置”的权限模式已经无法适应自然语言交互、跨系统连接器调用和长链路自动化执行场景。企业需要的不只是一个自动填表工具，而是一套“给 AI 发通行证”的机制，使 Agent 能够在被授权的边界内代替员工完成申请、执行和治理。

## 3. 产品定位

本产品定位为一套面向企业权限申请场景的 Agent 身份与权限治理产品。

它不是单纯的审批流工具，也不是传统 IAM 的替代品，而是位于“用户业务意图”和“企业底层权限系统”之间的一层智能治理中台，负责完成：

- 将自然语言需求翻译为可执行的最小权限申请。
- 为 Agent 建立独立身份与委托边界。
- 根据任务状态、风险等级和上下文自动决定审批和开通路径。
- 管理权限的生效、续期、到期和回收。
- 为企业后续扩展到多 Agent、多系统、多连接器提供统一治理框架。

## 4. 产品愿景与目标

### 4.1 产品愿景

让企业从“人找系统要权限”升级为“人告诉 Agent 业务目标，系统按最小权限原则自动发放、管理和回收通行证”。

### 4.2 业务目标

- 将权限申请平均处理时长从 3 到 5 天降低到 30 分钟以内。
- 将权限相关人工沟通成本降低 80%。
- 降低因配置错误导致的误授权和重复开通。
- 建立可复用的 Agent 身份与权限治理能力，为后续多系统接入铺路。

### 4.3 产品目标

- 支持员工通过自然语言提出权限需求。
- 支持系统自动识别资源、动作、时效和最小权限建议。
- 支持按风险等级自动发起审批和升级审批。
- 支持审批通过后自动调用外部系统完成开通。
- 支持到期提醒、一键续期和自动回收。
- 支持完整的审计和责任追踪。

## 6. 核心问题定义

### 6.1 当前问题

当前权限申请存在以下断点：

1. 用户用业务语言表达需求，但系统需要的是权限语言。
2. 审批流只判断“要不要给”，不判断“应该给多大、给多久、是否需要升级审批”。
3. 开通动作与审批动作割裂，无法形成闭环。
4. 权限开通后缺少生命周期治理，过期和续期依赖人工跟踪。
5. Agent 参与流程后，缺少“Agent 替谁办事、凭什么办、边界在哪里”的机制。

### 6.2 产品要解决的问题

- 让用户不用理解底层权限模型，也能准确表达申请意图。
- 让系统自动将业务意图转为最小权限申请。
- 让审批人看到面向业务的解释和风险提示。
- 让批准结果能够直接驱动权限开通，而不是再次依赖人工翻译。
- 让权限天然具备时效、提醒、回收和续期能力。
- 让 Agent 的身份、委托和行为边界可被治理和审计。

## 7. 核心场景

### 7.1 主场景

员工说：“我需要能查看销售部 Q3 的报表，但不需要修改权限。”

产品需完成：

1. 理解用户想看什么、能做什么、不能做什么。
2. 判断对应的最小权限范围，例如只读 scope。
3. 判断是否跨部门、是否高敏、是否需要升级审批。
4. 发起审批。
5. 审批通过后自动开通。
6. 记录有效期。
7. 到期前提醒续期，到期后自动回收。

### 7.2 高频扩展场景

- 用户申请某文档或报表的编辑权限。
- 用户申请临时访问某敏感数据集。
- 项目成员申请共享目录短期访问权限。
- 员工申请续期已有权限。
- 安全管理员紧急撤销某 Agent 或某类授权。

### 7.3 失败与异常场景

- 用户表达不清，无法准确识别资源或动作。
- 用户没有资格申请该资源。
- 审批通过但系统开通失败。
- Agent 被停用但外部 token 尚未失效。
- 续期时资源风险等级发生变化。

## 8. 产品原则

### 8.1 最小权限原则

系统默认只授予完成当前任务所需的最小权限，不允许按“大概够用”放大授权。

### 8.2 默认可追溯原则

每一次申请、审批、开通、续期、撤销都必须可追溯到用户、Agent、审批链和系统动作。

### 8.3 默认安全原则

当身份、委托、审批、策略或连接器状态不可判定时，系统默认不放行高风险动作。

### 8.4 业务语言优先原则

面向员工和审批人展示的内容优先使用业务语言，而不是底层权限项语言。

### 8.5 生命周期治理原则

权限不是一次性发完就结束，而是从申请开始就带有时效、提醒、续期和回收机制。

## 11. 用户故事

### 11.1 普通员工

作为普通员工，我希望直接告诉系统“我要看什么、做什么”，而不是研究具体权限项，这样我能更快拿到正确权限。

### 11.2 审批人

作为审批人，我希望看到申请背后的业务意图、资源敏感度和风险等级，而不是一串技术术语，这样我能更快做出判断。

### 11.3 IT 管理员

作为 IT 管理员，我希望审批通过后系统直接自动开通权限，避免我手工查 scope 和手工配置。

### 11.4 安全管理员

作为安全管理员，我希望高风险申请自动升级审批、到期自动回收，并能随时撤销某个 Agent 的权限影响范围。

## 12. 详细功能需求

### 12.1 自然语言申请

#### 功能目标

让员工用业务语言表达权限需求。

#### 功能说明

- 支持员工输入自然语言描述。
- 支持识别“资源对象、动作、限制条件、时效偏好、业务理由”。
- 支持系统在意图不明确时发起澄清。

#### 产品规则

- 若用户明确表达“只看不改”，系统默认建议只读权限。
- 若用户未填写时效，系统按默认有效期生成建议。
- 若系统无法识别资源或动作，不能直接进入开通流程。

#### 成功标准

- 用户输入后可生成结构化申请单。

### 12.2 最小权限建议

#### 功能目标

把业务语义翻译成最小权限建议。

#### 功能说明

- 根据资源类型和动作类型生成最小权限项。
- 支持展示“为什么推荐这个权限”。
- 支持限制性条件，例如禁止修改、禁止分享、禁止导出。

#### 产品规则

- 优先生成资源级而非系统级权限。
- 优先生成只读而非编辑权限。
- 若存在多种可行权限组合，优先推荐影响面最小的组合。

### 12.3 风险分级

#### 功能目标

对每一次申请进行风险识别，并决定审批链。

#### 功能说明

- 综合资源敏感度、动作危险度、授权时长、组织边界和副作用进行评分。
- 将申请划分为低风险、中风险、高风险。
- 将风险结果展示给审批人和管理员。

#### 产品规则

- 高敏资源、写权限、跨部门访问、长期授权应提升风险等级。
- 任意一项达到高风险阈值时，整体按高风险处理。

### 12.4 动态审批

#### 功能目标

根据风险等级自动决定审批链，减少不必要的人工摩擦。

#### 功能说明

- 低风险可自动批准或简化审批。
- 中风险走直属主管与资源 Owner。
- 高风险增加安全管理员审批。

#### 产品规则

- 审批通过不等于权限已生效，仍需等待系统实际开通。
- 审批撤回或超时后，流程不得继续推进。

### 12.5 自动开通

#### 功能目标

让审批结果直接驱动系统权限开通。

#### 功能说明

- 审批通过后自动调用目标系统连接器。
- 返回开通状态并通知用户。
- 记录实际生效时间和过期时间。

#### 产品规则

- 若连接器调用失败，系统状态应显示“已批准未生效”，而不是“已开通”。
- 开通动作必须二次校验资源、动作、审批和时效是否匹配。

### 12.6 生命周期治理

#### 功能目标

让权限从发放开始即进入可治理状态。

#### 功能说明

- 权限生效后记录有效期。
- 到期前自动提醒。
- 支持用户一键续期。
- 到期自动回收。

#### 产品规则

- 续期不是简单延长，而是要重新校验风险等级和上下文变化。
- 自动回收必须有可审计记录。

### 12.7 Agent 身份与委托治理

#### 功能目标

明确 Agent 替谁办事、能办到什么程度。

#### 功能说明

- 为权限助手 Agent 建立独立身份。
- 用户在发起代办时形成委托关系。
- 委托范围包括资源、动作、时效和再委托限制。

#### 产品规则

- 无委托时，Agent 不得代用户申请权限。
- Agent 停用后，其已建立会话和后续执行应被中断。

### 12.8 审计与追踪

#### 功能目标

让业务、技术和安全三方都能回答“谁做了什么、为何被允许”。

#### 功能说明

- 记录申请、审批、开通、续期、撤销和回收全流程。
- 支持按用户、资源、Agent 和时间检索。
- 支持异常场景追溯。

#### 产品规则

- 关键链路不得出现无日志动作。
- 高风险动作必须记录裁决理由。

## 13. 关键流程

### 13.1 主流程

1. 用户输入自然语言需求。
2. Agent 识别需求并生成结构化申请。
3. 系统生成最小权限建议。
4. 系统进行风险分级。
5. 系统发起对应审批流。
6. 审批通过后系统自动开通权限。
7. 系统通知用户开通结果。
8. 权限进入到期提醒、续期或自动回收流程。

### 13.2 续期流程

1. 系统提前触发到期提醒。
2. 用户点击续期。
3. 系统重新评估风险和资源状态。
4. 如需审批则重新发起审批。
5. 续期成功后更新过期时间。

### 13.3 撤销流程

1. 用户、管理员或系统触发撤销。
2. 系统更新授权状态。
3. 系统调用连接器执行撤销。
4. 系统回写撤销结果并记审计日志。

## 14. 页面/交互视角需求

### 14.1 员工侧

- 输入自然语言申请。
- 查看解析后的权限建议。
- 查看审批状态和开通状态。
- 接收到期提醒。
- 发起一键续期。

### 14.2 审批侧

- 查看业务意图。
- 查看资源、动作、时效。
- 查看风险等级和原因。
- 做出同意、拒绝或补充说明。

### 14.3 管理员侧

- 查看 Agent 身份和状态。
- 查看风险规则和审批流配置。
- 查看授权生效情况和异常列表。
- 发起紧急撤销。

## 15. 业务规则

### 15.1 授权粒度

最小授权单位为“资源 + 动作 + 时效”。

### 15.2 默认有效期

V1 默认：

- 普通只读权限：7 天。
- 高风险权限：24 小时到 3 天。
- 长期权限：不作为默认选项，必须显式审批。

### 15.3 审批规则

- 低风险：自动批准或直属主管审批。
- 中风险：直属主管 + 资源 Owner。
- 高风险：直属主管 + 资源 Owner + 安全管理员。

### 15.4 续期规则

- 续期需复用历史记录，但必须重新校验上下文变化。
- 若资源敏感度提升，必须升级审批。

### 15.5 撤销规则

- 用户离职、Agent 停用、委托失效、任务结束、审批撤回均可触发撤销。

## 17. 竞品/替代方案思考

当前企业常见替代方案包括：

- 表单 + OA 审批 + 人工 IT 开通。
- 单点 IAM 工具中的权限申请模块。
- 自定义机器人辅助填单。

与这些方案相比，本产品的核心差异在于：

- 支持自然语言到权限建议的转换。
- 将 Agent 纳入身份与委托治理。
- 将审批、开通、续期、回收做成统一闭环。
- 提供面向后续多系统、多 Agent 扩展的产品基础。

## 18. 版本规划

### 18.1 V1

- 权限自助服务 Agent。
- 飞书审批接入。
- 飞书文档/报表类只读权限开通。
- 风险分级、审批升级、续期与回收。
- 审计日志与管理员视图。

### 18.2 V1.5

- 增加编辑类权限治理。
- 增加更多资源类型。
- 增加异常处理和补偿可视化。

### 18.3 V2

- 支持多系统连接器。
- 支持更丰富的 Agent 身份治理。
- 支持更细粒度策略和统一会话治理能力。

## 19. 风险与依赖

### 19.1 项目风险

- 用户自然语言表达过于模糊，导致解析不准确。
- 外部系统权限模型复杂，难以快速建立最小权限映射。
- 外部审批或权限接口回调不稳定。
- 高风险权限场景中，企业现有审批流程可能不统一。

### 19.2 关键依赖

- 企业 IAM/SSO。
- 资源目录和资源 Owner 信息。
- 飞书审批和飞书权限接口可用性。
- 安全和 IT 团队配合梳理最小权限映射表。

## 20. 成功判定

产品 V1 上线成功的标准包括：

- 员工能够通过自然语言提交权限申请。
- 系统能够自动生成结构化权限申请和最小权限建议。
- 中低风险申请能显著缩短处理时间。
- 审批通过后的权限可自动开通。
- 权限到期提醒、续期、回收可闭环运行。
- 高风险场景下具备明确的升级审批和审计能力。
